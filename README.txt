Overview
README file for “The Protective Effects of a Healthy Spouse: Medicare as the Family Member of Last Resort,”* forthcoming in XXX.

Notes: This README follows the template endorsed by the AEA (American Economic Association).
The code in this replication package constructs the analysis dataset from raw Medicare administrative records using Stata and R, and then produces all the empirical results (tables and figures) presented in the paper (including the main text and the online appendices). Specifically, the Stata scripts process the CMS Medicare claims data (identifying patient spouses and health events) and generate the reduced-form evidence (event-study graphs and regression results). The R code implements a dynamic model simulation and counterfactual policy analysis described in the paper’s Appendix B.
Estimated runtimes: The data construction phase is computationally intensive and is best run on a high-performance computing environment. Constructing the analytical sample from raw Medicare files can take on the order of 12–24 hours (this can be parallelized by year for efficiency). The reduced-form analysis code (Stata) can be executed in roughly 6–10 hours once the dataset is prepared, given a modern server. The dynamic model estimation (R) is more computationally demanding; it may require multiple days to converge, depending on processing power and parallelization.

Data Availability and Provenance Statements
This study uses administrative health care data from the U.S. Medicare program, including Medicare claims for hospital and post-acute care and the Minimum Data Set (MDS) assessments for nursing home stays. The data span the population of Medicare beneficiaries and include de-identified individual records on health care utilization such as hospitalizations, skilled nursing facility (SNF) stays, and other health services. A novel feature of the dataset is the identification of spousal pairs: Medicare beneficiaries are linked into couples by matching addresses on record with the Centers for Medicare & Medicaid Services (CMS). We identify opposite-sex Medicare recipients living at the same address (with at most two such recipients per address) as spouses (a method validated by Gross et al. 2024, as cited in the paper). This linked dataset enables the analysis of within-household spillovers in formal care use. The dataset was obtained through a restricted-use Data Use Agreement (DUA) with CMS and is not publicly available due to privacy and confidentiality laws (HIPAA) that protect personal health information.
Researchers interested in accessing similar data may request access through CMS’s data access channels. In practice, this typically requires an application via the CMS Research Data Assistance Center (ResDAC) or the CMS Virtual Research Data Center (VRDC). Access to Medicare claims data requires institutional review and approval, a signed DUA, and adherence to strict data security protocols. Researchers will likely need to demonstrate an IRB (ethics board) approval or exemption for use of health data, and there are substantial costs and processing times associated with obtaining these data. These data cannot be shared by the authors under the DUA terms; however, all programs used to construct and analyze the data are provided so that other researchers with appropriate data access can replicate the results. The spousal crosswalk is not typically included in these DUAs.

Statement about Rights
We certify that the authors of the manuscript had legitimate access to and permission to use the confidential Medicare data used in this study. All data use was in accordance with the terms of the DUA with CMS and applicable laws. The authors do not have the rights to redistribute the raw or derived individual-level data, which is why these data are not included in the public replication package.
Computational and Software Requirements
•	Automated setup: The user must configure paths and install required libraries manually, as described below.
•	Stata – The Stata code was last run with Stata 15 (Stata SE 15.1). It should be compatible with Stata versions 15 and above (including Stata 16, 17, etc. released 2017–2025). All Stata .do files can be executed on 64-bit Stata; no unusual or user-written Stata packages are required beyond base Stata.
•	R – The dynamic model code was run with R 4.4.1. The following R packages (CRAN libraries) are required: tidyverse, gmm, EnvStats, broom, purrr, furrr, pracma, Rcpp, future, and base parallel. Please ensure these libraries are installed in R 4.4.1 (or a later version). The R code will attempt to load these libraries; if not present, install them via install.packages("<package-name>") before running.
•	Portions of the analysis use C++ code through the Rcpp package. Thus, a suitable C++ compiler is required (for example, Rtools on Windows or the GCC compiler on Linux/Mac). The provided C++ source files will be compiled at runtime by R (via Rcpp::sourceCpp in the R scripts).
•	Controlled randomness: Random number generation is controlled for reproducibility. A fixed random seed is set in the dynamic model code (see 1_DynamicModel_Estimation.Rmd, which uses set.seed(20250212)). The Stata analyses are purely deterministic (they do not involve random sampling/bootstrapping, so no Stata seed is needed).

Hardware: Due to the volume of Medicare data, replication requires a high-memory computing environment. We recommend at least 128 GB of RAM for processing the raw data comfortably; the data construction step may not be feasible on a typical desktop. The analysis was originally performed on a server; attempting to run it on a personal computer with insufficient memory will likely fail. Adequate disk storage is also needed to host the raw data (hundreds of gigabytes) and intermediate files.

Description of Programs and Code Structure
The replication package is organized with a master control script and a set of subdirectories containing Stata and R programs. Below we describe the contents and how they correspond to the paper’s analyses:
•	Master script: ValueHealthySpouse_Master.do – This is the primary master do-file that orchestrates the replication. It defines global paths and calls the sequence of Stata programs needed to replicate the study. Running this script (after configuring paths) will execute the data construction and analysis code in the correct order for the main paper results and appendices (except the dynamic model R code, which is run separately in R).
•	Stata code for data construction and analysis (located in the ValueHealthySpouse_Replication/ directory):
    •	1_IdentifyIndexEvents.do – Identifies index health shock events for the analysis. This program scans the Medicare inpatient claims (MEDPAR files) to find all hospitalizations for heart attack or stroke that will serve as the spouse’s “shock” event. It applies filters (e.g. excluding events that are SNF admissions or repeat events as needed) and creates an output dataset of shock events with details (beneficiary ID, admission date, diagnosis codes, discharge status, etc.).
    •	2_ResponseEvents_MEDPAR.do – Gathers partner response events from Medicare claims. For each identified shock event, this program pulls the outcome spouse’s subsequent utilization of formal care from the MEDPAR data (inpatient/skilled nursing claims). It flags instances such as admissions to a skilled nursing facility (SNF) for the outcome spouse after the partner’s shock.
    •	2a_ResponseEvents_ED.do – Gathers the outcome spouse’s Emergency Department visits from the Medicare outpatient/emergency claims, to measure another aspect of health care use following the partner’s shock.
    •	2b_ResponseEvents_mortality.do – Incorporates mortality outcomes for the outcome spouse. This program uses Medicare enrollment data (or mortality information in claims) to flag if and when the outcome spouse dies, which is important for analyses conditioning on survival.
    •	2c_ResponseEvents_Falls.do – Extracts information on falls-related hospitalizations for the outcome spouse. It identifies inpatient stays specifically for injury or fall diagnoses as another health outcome of interest after the partner’s health shock.
    •	3_ConstructPanel.do – Merges and constructs the panel dataset for analysis. This program takes the outputs of steps 1 and 2 (shock events and various outcome events) and combines them into a panel structure at the household level. It creates a timeline (relative to the shock event) for each couple, indicating the outcome spouse’s monthly status (e.g., in SNF or not) and other outcomes over time. This is the main analysis dataset used for the event-study regressions.
    •	3a_SummaryStatsTable.do – Produces Summary Statistics for the sample (Table 1 in the paper). It uses the constructed dataset to calculate descriptive statistics (e.g., mean age, distribution of chronic conditions, etc. for shock spouses and outcome spouses) and outputs the results in a format for the table.
    •	4_MainEventStudy.do – Runs the core event-study difference-in-differences analyses for various outcomes. This program is parameterized to take an outcome type (such as SNF use) and a sample restriction (e.g., balanced panel or not) and then estimates equation (1) from the paper, which tracks the outcome spouse’s outcome before and after the shock spouse’s event. It produces the estimates for the event-study graphs (e.g., coefficients δ<sub>n</sub> for each month relative to the event). This general code is used for multiple figures by varying the outcome and sample.
    •	4a_MainEventStudy_snf-entry.do – A companion to the above, focused on SNF entry outcomes. For the analysis of entering a skilled nursing facility (as opposed to being in one), this script computes event-study results specifically for the probability that an outcome spouse enters an SNF in a given month. (Figure 1 of the paper has two panels: (a) indicator for being in SNF, and (b) indicator for SNF entry; those are generated by 4_MainEventStudy.do and 4a_... respectively.)
    •	4b_MainEventStudy_FatalitySplit.do – Performs a heterogeneity analysis on the event study, splitting by whether the shock event was fatal for the shock spouse. This generates the results for Figure 2 in the paper, comparing the outcome spouse’s SNF utilization trajectory in cases where the shock spouse survives vs. cases where the shock spouse dies from the heart attack/stroke.
    •	5_HealthEffectsTable.do – Estimates the effect of a spousal health shock on various health outcomes for the outcome spouse, for Table 2 in the paper. Specifically, this program runs difference-in-differences regressions (per equation (2) in the paper) for outcomes such as any inpatient admission, falls-related admission, emergency department visit, etc. It is designed to be run for each outcome category (the program uses arguments or global settings to switch the outcome variable). Each run produces the point estimate and standard error for one column of Table 2.
    •	5a_MakeTable.do – Gathers and formats the results from the above regressions into the final Table 2 output. After 5_HealthEffectsTable.do is run for all required outcomes, this script assembles the coefficient estimates into a single table (likely producing a LaTeX .tex file or an output table for publication).
    •	6_MortalityEffects.do – Conducts an event-study analysis for outcome spouse mortality. This corresponds to Figure 3 in the paper. The program calculates how the partner’s death probability evolves following the shock spouse’s event (essentially estimating an event-study where the outcome is an indicator that the outcome spouse died). It accounts for censoring as needed.
    •	7a_SplitEffects_ChronicCondition.do – Runs event-study regressions split by the outcome spouse’s pre-existing chronic condition status. This produces Figure 5 in the paper, which shows how the effect on SNF usage differs for outcome spouses who had a chronic health condition versus those who were relatively healthy prior to their partner’s shock.
    •	7b_SplitEffects_IndexDischarge.do – Runs event-study analyses split by the severity of the shock spouse’s health event, as proxied by discharge status or length-of-stay. This script generates Figure 6 in the paper, examining if the spouse’s incapacitation level (for instance, whether the shock spouse had a long hospital stay or complications) affects the outcome spouse’s increase in formal care use.
    •	8_AnnualSpending.do – Analyzes the impact of the spousal shock on annual Medicare spending for the outcome spouse (difference-in-differences at the annual level). This produces the results in Table 3 of the paper. It aggregates outcomes by year (relative to the shock) to estimate how the partner’s total Medicare-covered expenditures change in the year of the event (and possibly the year after). The program generates regression results for total spending and possibly sub-categories of spending, as discussed in Section 3.1 of the paper.
    •	9_CostSharingRD.do – Implements the regression discontinuity (RD) design to study price sensitivity to formal care. Medicare fully covers the first 20 days of an SNF stay, after which copayments kick in starting on day 21. This script analyzes how outcome spouses respond to that cost-sharing threshold, and how that response differs before vs. after their partner’s shock. It uses an RD approach (at day 20 of SNF stay length) to estimate the drop-off in continuation of SNF care once it becomes costly, comparing households with healthy vs. incapacitated spouses. The output feeds into Table 4 (which shows changes in price sensitivity due to the shock) and also into Figure 8.
    •	9a_CostSharingRD_ChronicCondition.do – An extension of the above RD analysis, splitting the sample by the outcome spouse’s chronic condition status. This generates the subgroup estimates by chronic condition shown in Table 4 (columns for “No chronic condition” vs. “Has chronic condition”). Essentially, it computes the RD treatment effect (the change in SNF utilization at day 20) for each subgroup, allowing us to see how the spouse’s health status affects price sensitivity.
    •	9b_CostSharingRD_EventStudy.do – Produces an event-study style figure of SNF stays beyond 20 days as the outcome. This script generates Figure 9 in the paper, which plots how the probability of having an SNF stay that exceeds 20 days (and thus incurs cost sharing) evolves after the spouse’s shock. It is similar to the earlier event studies but focuses on the indicator that the outcome spouse’s SNF use went beyond the free 20-day period.
•	Appendix analysis code (in the ValueHealthySpouse_Replication/Appendix/ subdirectory): This folder contains additional Stata do-files that produce the results presented in the paper’s Appendix (denoted as Appendix Figures or Tables). These programs typically mirror or extend the main analyses for robustness and subgroup checks. For example:
    •	Appendix_Robustness_No2YearRequirement.do relaxes a sample restriction (e.g., the requirement that households have no additional shocks within 2 years) and re-estimates the main event study (Appendix Figure A1).
    •	Appendix_Circularity.do checks for “circular” causation (whether the measured shock might be influenced by prior outcomes spouse events) by varying definitions or sample (producing Appendix Figure A2).
    •	Appendix_MainEventStudy_gendersplit.do runs the main event-study separately for male vs. female outcome spouses (Appendix Figure A3, gender heterogeneity).
    •	Appendix_Eventstudy_agesplits.do does the event-study for different age groups of outcome spouses or shock spouses (Appendix Figure A5, age heterogeneity).
    •	Appendix_SplitEffects_Race.do, Appendix_SplitEffects_IndexLOS.do, Appendix_SplitEffect_TimeofDeath.do, etc., each correspond to additional heterogeneity analyses described in Appendix Section A (such as splits by race, by the shock spouse’s length-of-stay, by timing of the outcome spouse’s death, etc.). These generate various Appendix Figures (A6, A7, A8, etc.).
    •	Appendix_ExtraHouseholdEvents_EventStudy.do examines other household events (e.g., if there were additional health shocks beyond the spouse, though the sample is restricted to one shock spouse) – used for a robustness check in Appendix Figure A4 or A9.
    •	Appendix_LocationPreferences.do explores whether outcome spouses tend to enter the same SNF as the shock spouse (a check for couples’ location decisions, Appendix Figure A4).
    •	LASSO_predict_snf.do and merge_LASSOdata.do – These two programs relate to a supplementary analysis using a machine learning approach (LASSO) to predict SNF utilization. The LASSO_predict_snf.do file develops a LASSO regression model to predict the probability that an outcome spouse will be discharged to a SNF given their current and past hospitalizations. It likely uses a separate dataset or the panel data to train this predictive model (this may involve creating variables for patient history, etc.). The merge_LASSOdata.do then merges the LASSO-generated predictions back into the main analysis dataset. This allows the authors to compare predicted SNF use to actual SNF use. The output of these programs is used for Figure 7 in the paper (which compares changes in actual SNF admission probability vs. changes in predicted probability in the absence of a healthy spouse).
•	Dynamic model code (in the ValueHealthySpouse_Replication/Appendix_DynamicModel/ subdirectory): This folder contains the R and C++ code for the structural model described in Appendix B of the paper. In this model, households face a dynamic decision problem regarding formal care use, and the model is calibrated to match the empirical moments. Key files include:
    •	1_DynamicModel_Estimation.Rmd – An R Markdown document that estimates/calibrates the dynamic model. It sets up the model parameters, uses the observed data moments, and runs a Generalized Method of Moments (GMM) estimation routine to fit the model to the data. This code uses the tidyverse for data manipulation and the gmm package for estimation; it also uses Rcpp to source C++ functions that simulate the model. (The output of this Rmd is primarily the estimated parameters and model fit diagnostics; it is not needed for main text tables, but it’s required to generate the counterfactual analysis that follows.)
    •	simulate_person.cpp and simulate_person_value.cpp – C++ source files that contain the simulation routines for the dynamic model. These implement the model’s logic (e.g., simulating a person’s sequence of care decisions or health states given parameters) and compute the value function or welfare for given policies. The R code compiles these using Rcpp to speed up the simulation within the estimation.
    •	ObservedMoments.csv – A CSV file containing the empirical moments (statistics) observed in the data that the dynamic model will target. These might include, for example, rates of SNF use under different scenarios (healthy vs. sick spouse) or price elasticity measures. By providing this file, the authors allow the dynamic model code to run without requiring re-computation of all moments from the raw data. (This is the only data file included in the package, and it contains aggregated non-confidential information.)
    •	2_MakeHeatmap_CoinsuranceSpace.R – An R script that generates a figure (heatmap or plot) of welfare gains across coinsurance rates for SNF care, differentiated by spousal health status. This corresponds to Figure 10 in the paper, which illustrates the counterfactual policy analysis: how optimal SNF coverage generosity should differ depending on whether one’s spouse is incapacitated. The script likely uses the results of the calibrated model to simulate household welfare under different SNF copayment levels for two scenarios (healthy spouse vs. sick spouse), and visualizes the welfare difference.
    •	2a_MakeHeatmap_OnsetSpace.R – An R script that produces an additional figure (likely an Appendix figure, possibly Figure B2) exploring a different dimension of the counterfactuals (perhaps varying the timing or likelihood of the spouse’s shock). It might create a similar heatmap for alternative scenario parameters.
•	Utility code: Inflation.do – A standalone Stata do-file that contains routines to adjust monetary values for inflation (e.g., converting spending figures to real dollars of a base year). Some analysis scripts (such as the spending analysis) call this to standardize financial amounts. This file includes inflation indices to deflate dollar values to a common year.

Note on directory paths: The .do files assume certain directory structures set via global macros at the top of the Master script (for example, an $input_datapath for raw data and an $allcode path for the code directory). The user will need to modify these to point to the correct locations on their system (see Instructions below). The output (tables, figures) will typically be saved in designated output folders or in the working directory as specified in the code (the code will create output files such as .dta, .log, or .tex files for results). If necessary, create subfolders (e.g., output/) or adjust output paths according to the code’s expectations.

Instructions to Replicators
How to run the replication: The following is a step-by-step guide to using the provided code and data to reproduce the paper’s results. Each step should be followed in sequence.
1.	Obtain the required data – Ensure you have access to the necessary Medicare data (inpatient claims, etc.) including the CMS MEDPAR files and MDS, with the ability to identify spouse pairs via the crosswalk. These should be organized as required (for example, in a directory with Medicare claims data by year). Due to data restrictions, the actual data files are not included in this package; you must have your own access through CMS.
2.	Adjust file paths in the code – Open ValueHealthySpouse_Master.do in Stata and edit the global path definitions at the top. You will need to specify the location of the raw data files and the working directories. For example, set ${input_datapath} to the folder where your Medicare data (e.g., MEDPAR datasets) reside, and if needed, adjust ${output_path} for results. Also confirm any other global macros (like ${allcode} which should point to the ValueHealthySpouse_Replication/ code directory if not already correctly set). Similarly, if running in a different environment, you may need to update file paths within some do-files (e.g., file paths in use or merge commands) to point to your local data.
3.	Install and prepare software – Make sure Stata and R (with the required libraries) are installed on your system. In R, install the packages listed above under Software Requirements if you haven’t already. Also ensure you have a C++ compiler available for Rcpp.
4.	Run the Master do-file in Stata – Launch Stata (version 15 or higher) and run ValueHealthySpouse_Master.do. This will execute the sequence of Stata programs automatically. It will first construct the analysis dataset (by running scripts 1, 2, 2a, 2b, 2c, 3) and then proceed to generate all main analysis results (scripts 3a, 4, 4a, 4b, 5, 5a, 6, 7a, 7b, 8, 9, 9a, 9b, etc., including those in the Appendix folder). The Master script is designed to produce all results for the paper in one go. Note: Given the heavy data processing, this may take many hours; you can monitor progress in the Stata log. If the process is too slow or memory-intensive, consider running the data construction portions (steps 1–3 above) separately or in parallel by year, as the code can be adapted to split the workload.
5.	Run the dynamic model code in R – The structural model is executed outside of Stata. Open the file Appendix_DynamicModel/1_DynamicModel_Estimation.Rmd in RStudio (or run it with an R Markdown-compatible workflow). Before running, set the working directory to the Appendix_DynamicModel folder (so that the code can find ObservedMoments.csv and write output files). You may also need to edit the Rmd: specifically, adjust the path variable defined near the top. Currently it may be pointing to an author-specific directory (e.g., a Dropbox path); you can replace this with your own directory or simply use path = "." (current directory) if appropriate. Then run all cells in the R Markdown. This will compile the C++ functions (using Rcpp) and perform the GMM estimation of the model. It will output estimated parameters and may save results internally for the next step.
6.	Generate model-based figures – After the model estimation is done, run the R scripts 2_MakeHeatmap_CoinsuranceSpace.R and 2a_MakeHeatmap_OnsetSpace.R (in R or RStudio). These will use the model’s results to produce the figures corresponding to the policy counterfactuals (Figure 10 in the paper and an additional Appendix figure). The scripts will likely save the figures (for example, as image files or plots in R). Ensure that any file paths in these scripts (if they read the model output or write images) are correctly pointing to your desired output location.

By following these steps, a replicator with access to the Medicare data should be able to reproduce all main findings of the paper. If issues arise (such as memory errors or missing libraries), please refer back to the computational requirements and ensure the environment is properly set up. For additional guidance, contact the authors (see contributor emails above) or consult the AEA Data Editor’s resources.
